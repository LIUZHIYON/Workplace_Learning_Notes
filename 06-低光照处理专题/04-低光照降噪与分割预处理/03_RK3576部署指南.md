# 03 - RK3576 部署指南

## 1. RK3576 平台概览

### 芯片架构

```
┌─────────────────────────────────────────────────┐
│                   RK3576                         │
├─────────────────────────────────────────────────┤
│  CPU Big: 4×Cortex-A72 @ 2.2GHz (性能核)        │
│  CPU LITTLE: 4×Cortex-A53 @ 2.0GHz (能效核)     │
│  GPU: Mali-G52 MC3 @ 1GHz, 145 GFLOPS           │
│  NPU: 6 TOPS (INT8), 支持 INT4/FP16/BF16         │
│  RAM: LPDDR4x/LPDDR5 32-bit                      │
│  NEON: ✅ 独立 NEON 协处理器 (128-bit SIMD)       │
│  ISP: 16MP, 3DNR, HDR 120dB, 鱼眼校正            │
├─────────────────────────────────────────────────┤
│  视频解码: 8K@30 / 4K@120                        │
│  视频编码: 4K@60 H.264/H.265                     │
│  显示: HDMI 2.1, eDP, MIPI DSI, DP 1.4          │
│  USB: USB 3.0 DRD ×2                             │
└─────────────────────────────────────────────────┘
```

### 性能定位

| 芯片 | CPU | DMIPS | NPU | 定位 |
|------|-----|:-----:|:---:|------|
| RK3588 | 4×A76 + 4×A55 | 93K | 6T | 旗舰 |
| **RK3576** | **4×A72 + 4×A53** | **58K** | **6T** | **中高端** |
| RK3568 | 4×A55 | 20K | 1T | 中端 |

RK3576 = RK3588 同级 NPU + 略逊的 CPU，非常有性价比。

### 任务分配策略

```
RK3576 任务分配
├── ISP (硬件):  3DNR 降噪、HDR、坏点校正 → 0 CPU 开销
├── OpenCV (CPU): 导向滤波 / 双边滤波 / CLAHE → 预留 10-20ms
├── NPU (RKNN):   图像分割模型推理 → 预计 50-100ms
└── 总延迟:       ~80-150ms (6-12 FPS)，适合实时
```

### 2.3 运行时 CPU 绑定

```cpp
// C++: 将 OpenCV 线程绑定到 A72 性能核
#include <sched.h>
cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(0, &cpuset); CPU_SET(1, &cpuset);
CPU_SET(2, &cpuset); CPU_SET(3, &cpuset);  // A72 核心
sched_setaffinity(0, sizeof(cpuset), &cpuset);

// OpenCV 线程数建议: 2-4
cv::setNumThreads(2);
```

**关键**: 给 NPU 推理留出 CPU 资源，不要用满 8 核。

## 3. OpenCV NEON 优化详解

### 3.1 哪些函数自动受益

OpenCV 编译时开启 NEON 后，以下函数 **自动获得 2-4× 加速**：

| 函数类别 | 示例 | NEON 加速效果 |
|----------|------|:---:|
| 颜色转换 | `cvtColor`, `split`, `merge` | ✅ 2-4× |
| 滤波 | `GaussianBlur`, `medianBlur`, `boxFilter` | ✅ 2-3× |
| 几何变换 | `resize`, `warpAffine` | ✅ 2-3× |

## 4. 预编译思路 — 静态 LUT

### 4.1 预计算 Gamma LUT

```cpp
// 编译期或初始化时计算一次
static cv::Mat gamma_lut;

void init_gamma_lut(float gamma = 0.65f) {
    gamma_lut = cv::Mat(1, 256, CV_8U);
    uchar* p = gamma_lut.ptr<uchar>();
    for (int i = 0; i < 256; i++) {
        p[i] = cv::saturate_cast<uchar>(255.0 * pow(i / 255.0, gamma));
    }
}

void apply_gamma(const cv::Mat& src, cv::Mat& dst) {
    cv::LUT(src, gamma_lut, dst);
}
```

### 4.2 预计算的 CLAHE 对象复用

```cpp
// 初始化一次，反复使用
static cv::Ptr<cv::CLAHE> g_clahe = nullptr;

cv::Ptr<cv::CLAHE> get_clahe() {
    if (!g_clahe) {
        g_clahe = cv::createCLAHE(1.5, cv::Size(8, 8));
    }
    return g_clahe;
}
```

## 5. 在 RK3576 上跑分割模型

### 5.1 典型端到端流程

```
 camera
   │
   ▼
 ISP (3DNR / HDR)
   │
   ▼
 V4L2 buffer ──► OpenCV Mat
   │
   ▼
【OpenCV 预处理】 (10-25ms)
  ├── 缩放 → 640×480
  ├── 导向滤波降噪
  └── LAB CLAHE 增强
   │
   ▼
 cv::dnn::blobFromImage / 自定义
   │
   ▼
 RKNN 推理 (50-100ms, NPU)
   │
   ▼
 后处理 + 结果显示
```

### 5.2 CPU 与 NPU 流水线并行

```cpp
// 伪代码: 双缓冲流水线
while (running) {
    // 帧 N: NPU 推理 (异步)
    // 帧 N+1: CPU 预处理 (并行)
    // 结果: 延迟 = max(CPU_time, NPU_time)
}
```

通过流水线并行，总延迟 = `max(预处理, NPU推理)` 而非 `预处理 + NPU推理`。

## 6. ISP 硬件降噪配置

⚠️ **最高优先级**: RK3576 内置 ISP 支持硬件 3DNR，**图像在进入 OpenCV 前已经过降噪**。

### 关键 ISP 参数

| ISP 功能 | 说明 | 建议配置 |
|----------|------|----------|
| **3DNR** | 时域降噪 (多帧融合) | 开启，强度 3-5 (范围 0-15) |
| **2DNR** | 空域降噪 | 开启，强度 2-3 |
| **HDR** | 高动态范围 (最高 120dB) | 低光场景开启 |
| **DPC** | 坏点校正 | 始终开启 |
| **BLC** | 黑电平校正 | 始终开启 |

### 通过 V4L2 控制 ISP

```cpp
// 配置 3DNR (具体路径因 SDK 而异)
#define ISP_3DNR_CTRL_ID 0x980000
struct v4l2_control ctrl;
ctrl.id = ISP_3DNR_CTRL_ID;
ctrl.value = 4;  // 强度值
ioctl(fd, VIDIOC_S_CTRL, &ctrl);
```

> **结论**: 如果 ISP 3DNR 配置得当，**OpenCV 端的降噪需求可以大幅降低**。可能只需要 CLAHE 增强即可，CPU 占用 < 5ms。

### 7.2 功耗估算

| 场景 | CPU 占用 | CPU 功耗 | NPU 功耗 | 总功耗 |
|------|:---:|:---:|:---:|:---:|
| 仅 CLAHE | 5% | ~0.1W | — | ~0.1W |
| 导向滤波 + CLAHE | 15% | ~0.3W | — | ~0.3W |
| 双边 + CLAHE | 25% | ~0.5W | — | ~0.5W |
| NLM + CLAHE | 50% | ~1.0W | — | ~1.0W |
| 任一种 + NPU 分割 | 15% | ~0.3W | ~2.0W | ~2.3W |

> RK3576 TDP ~5W，以上组合均有余量。
